# Skill 交叉评估流程

本流程改编自上游 [`sunxingboo/skill-evaluator`](https://github.com/sunxingboo/skill-evaluator) 的多评估者交叉验证方法。它用于降低单次主观评分偏差，不替代真实任务测试。

## 使用条件

- 用户明确要求多模型、交叉验证或多评估者评审。
- 发布影响较大，单评估者结论不足以支持决策。
- 运行环境可以启动至少两个彼此独立的评估者。

若只能由同一模型串行扮演多个视角，应标为“多视角敏感性分析”，不得声称是独立多模型证据，也不得因此提高 E0-E4 证据等级。

## 流程

1. **独立评分**：至少两个评估者使用同一份 Skill 材料和固定 8 维口径，分别输出各维分数、定位证据和问题。不得向评估者泄露预期分数或其他评估结果。
2. **分歧识别**：找出任意评估者分差至少 `2.0` 的维度。小于 `2.0` 的差异只记录，不强制互审。
3. **交叉审查**：评估者仅针对显著分歧审查对方证据，指出误读、漏读或评分锚点使用错误。
4. **仲裁**：由未参与初始评分的评估者或主 Agent 依据原始文件和互审论据裁定。
5. **汇总**：记录每维最终分数、共识度和仲裁理由，再计算固定 8 维总分。

## 仲裁规则

- `一致`：所有评分分差不超过 `1.0`，取平均值并保留一位小数。
- `多数`：多数评分接近、少数明显偏离，采用多数评分的平均值，并说明排除偏离值的证据。
- `仲裁`：没有明确多数时，采用证据最具体、与评分锚点最一致的结论，不机械取平均。

## 证据边界

- 多评估者一致只说明设计评分较稳定。
- 多评估者评分不能证明 Skill 已正确触发、真实任务已完成或目标平台兼容。
- E2 及以上仍要求可重复的实际运行记录；E3 仍要求目标平台验证。
- 最终报告必须说明评估者、模型或视角、执行方式、失败/降级情况和显著分歧。

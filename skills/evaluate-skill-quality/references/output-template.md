# 评估输出模板

输出必须先给结论，再给证据。不用空泛的“整体不错”代替准入判断。

## 1. 用户决策摘要

```text
准入建议：[A 建议准入 / B 有条件准入 / C 修改后复评 / D 不建议准入]
总分：[0-100]
目标范围：[个人 / 团队 / 公开]
硬门槛：[未命中 / 命中项]
核心结论：[一句话]
最大风险：[一句话]
准入前必做：[最多 3 项]
```

## 2. D1-D9 评分表

| 维度 | 分数 | 权重得分 | 证据 | 主要缺口 |
|---|---:|---:|---|---|
| D1 | x/5 | x/12 | 文件/行/测试 | ... |
| ... | ... | ... | ... | ... |
| D9 | x/5 | x/14 | 文件/行/测试 | ... |

在表前或表后明确写：“D1-D8 来自原有八维框架；D9 是本评估流程增加的可验证性扩展。”

## 3. 触发与工作流测试

| 类型 | 样例 | 预期 | 结果 | 证据状态 |
|---|---|---|---|---|
| 应触发 x3 | ... | 触发 | ... | 已执行/仅设计 |
| 不应触发 x3 | ... | 不触发 | ... | 已执行/仅设计 |
| 正常流程 x2 | ... | 完成交付 | ... | 已执行/仅设计 |
| 失败/边界 x1 | ... | 明确停止或降级 | ... | 已执行/仅设计 |

## 4. 给 Agent 的优化任务

每项都必须可直接执行和验收：

| 优先级 | 问题 | 修改文件 | 具体动作 | 验收方法 | 不修的后果 |
|---|---|---|---|---|---|
| P0/P1/P2/P3 | ... | `path` | ... | 命令/case/输出 | ... |

禁止使用“优化一下、补充细节、增强鲁棒性”这类无法验收的任务描述。

## 5. 复评条件

列出下一次评估所需的明确证据，例如：

- `quick_validate.py` 通过。
- 3 个应触发和 3 个不应触发样例结果符合预期。
- 两个正常 case 产生约定交付物。
- 一个失败 case 正确停止，不伪造成功。
- 敏感信息和失效引用扫描无问题。

最后写清“达到什么条件可从 C 升为 B，或从 B 升为 A”。

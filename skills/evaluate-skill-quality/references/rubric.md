## 固定 8 维设计评分

| 编号 | 维度 | 权重 | 主要检查点 |
|---|---|---:|---|
| D1 | 元数据质量 | 15% | name 是否稳定明确；description 是否包含能力、触发场景和反向排除；是否会与相邻 Skill 冲突 |
| D2 | 执行引导清晰度 | 15% | 触发后是否知道先做什么、何时追问、如何选择分支、何时停止、何时请求确认 |
| D3 | 领域知识密度 | 15% | 是否包含通用模型不具备的规则、业务口径、决策标准或高价值经验；是否只是常识改写 |
| D4 | 工作流完整性 | 15% | 是否覆盖准备、执行、验证、异常、清理和迭代；复杂流程是否有状态和失败分类 |
| D5 | 输入输出清晰度 | 10% | 必需/可选输入、缺失处理、输出结构、证据字段、边界和降级结果是否明确 |
| D6 | 资源利用 | 10% | references/scripts/assets 是否按需组织；确定性逻辑是否下沉；是否避免把所有材料塞入 SKILL.md |
| D7 | 写作质量 | 10% | 指令是否可扫描、具体、一致、无冲突和无重复；是否控制上下文成本 |
| D8 | 范围与聚焦 | 10% | 是否解决一类稳定任务；边界是否明确；是否过宽、过碎或与其他 Skill 重叠 |

### D1 元数据质量

| 分数 | 标准 |
|---:|---|
| 0-3 | 元数据缺失、模糊或仍是占位文本 |
| 4-6 | 描述功能，但缺少触发条件 |
| 7-8 | 清楚描述功能和使用场景 |
| 9-10 | 同时给出精确能力、正向触发和反向排除条件 |

### D2 执行引导清晰度

| 分数 | 标准 |
|---:|---|
| 0-3 | 加载后仍不知道如何执行，缺少具体指令 |
| 4-6 | 有基本步骤，但缺少不同输入下的分支 |
| 7-8 | 不同场景有清晰路径，并说明何时追问 |
| 9-10 | 分支、执行边界和异常处理完整 |

### D3 领域知识密度

| 分数 | 标准 |
|---:|---|
| 0-3 | 没有专业知识，只是包装基础指令 |
| 4-6 | 有少量领域指导或约定 |
| 7-8 | 有丰富专家知识、数据模型或最佳实践 |
| 9-10 | 有难以从通用模型获得的深度知识和细致决策逻辑 |

### D4 工作流完整性

| 分数 | 标准 |
|---:|---|
| 0-3 | 没有工作流，只有单个操作或模糊指令 |
| 4-6 | 有部分步骤，但缺少异常处理或分支 |
| 7-8 | 多步骤工作流完整，顺序清楚 |
| 9-10 | 包含分支、异常处理和迭代循环的健壮工作流 |

### D5 输入输出清晰度

| 分数 | 标准 |
|---:|---|
| 0-3 | 输入和输出都未说明 |
| 4-6 | 只描述一部分，需要用户自行推断 |
| 7-8 | 输入要求、输出格式和示例清楚 |
| 9-10 | 有明确输入输出契约、多个示例和边界情况 |

### D6 资源利用

| 分数 | 标准 |
|---:|---|
| 0-3 | 明显需要资源但没有提供 |
| 4-6 | 有资源但利用不足或组织混乱 |
| 7-8 | 脚本、参考资料和资产分工合理 |
| 9-10 | 渐进式披露良好，SKILL.md 精简，配套资源完整 |
| N/A | 确实不需要附带资源时，按 7 分基线计入总分 |

### D7 写作质量

| 分数 | 标准 |
|---:|---|
| 0-3 | 组织混乱、表述模糊或指令冲突 |
| 4-6 | 可以阅读，但结构或简洁性不足 |
| 7-8 | 组织良好、标题清楚、表达直接、逻辑顺畅 |
| 9-10 | 可快速扫描、精确且没有冗余 |

### D8 范围与聚焦

| 分数 | 标准 |
|---:|---|
| 0-3 | 过宽而试图包揽一切，或过窄而不值得封装为 Skill |
| 4-6 | 范围基本合理，但有功能蔓延或边界不清 |
| 7-8 | 聚焦于单一且连贯的问题域 |
| 9-10 | 范围精确，并明确列出不处理的事项 |

等级按阈值依次判断：`S >= 9.0`，`A >= 7.5`，`B >= 6.0`，`C >= 4.0`，否则为 `D`。

S/A/B/C/D 只表示固定 8 维的设计等级，不直接表示已测试或可上线。每个分数必须引用具体文件内容；没有证据时不得仅凭印象给高分。

## 动态业务维度

动态维度不进入固定总分。每项包含：

```text
id: B1
name:
source:
critical: true/false
acceptance_criteria:
test_method:
evidence:
result: PASS/PARTIAL/FAIL/NOT_TESTED/BLOCKED/NOT_ASSESSED
```

生成原则：

- 从明确的业务需求、JD、验收规范或用户目标提取。
- 优先选择决定结果可用性的维度，不把写作风格拆成大量伪维度。
- 每项必须可观测、可重复、可判定。
- 同一业务目标下才用动态维度比较多个 Skill。

## 六道工程门

### G1 STRUCTURE

- 目录和依赖可解析。
- 主流程、详细规则、脚本和资产分层合理。
- 版本变更有可追踪记录；若无 CHANGELOG，至少能从版本控制定位修改和回归范围。

### G2 INPUT

- 关键输入、权限、环境和用户确认有显式检查。
- 条件不足时停止并返回缺口，不生成虚假确定结论。
- 高风险输入校验优先使用确定性代码或工具结果。

### G3 EXECUTION

- 工具调用顺序、状态变化和副作用明确。
- 超时、401/403/404/5xx、空结果和部分成功有处理。
- 重试有上限，危险动作有人确认，失败后可清理或回滚。

### G4 OUTPUT

- 主结果具有稳定 schema，包含证据、风险和待确认项。
- 图片、图表和附件等增强产物失败时，主结果仍可交付。
- 工具与证据收集结束后一次性生成最终结果，避免后续工具调用使正文丢失。

### G5 COMPATIBILITY

- 在目标平台检查 Markdown、附件、图片、字体、长文本和消息保留。
- 验证平台特有权限、沙箱、路径、编码和工具生命周期。
- 不用单平台成功推断所有平台可用。

### G6 REGRESSION

- 覆盖标准输入、缺失输入、超长输入、多对象、工具失败和目标平台。
- 包含应触发/不应触发样本。
- 历史 badcase 修复后进入回归集，并记录结果和证据。

“评测闭环 / 可验证性”在本体系中通过 G6、最低测试集、证据等级和新旧版对照落地，不作为 D9 进入固定加权总分。

## 证据规则

证据强度从低到高：

```text
作者声明 < 静态文件 < 单次演示 < 可重复测试日志
< 目标平台测试 < 多轮回归/灰度/线上指标
```

以下内容不能单独证明通过：

- SKILL.md 写了“必须验证”。
- AI 自己评价输出很好。
- 只有成功截图，没有输入、日志和失败样本。
- 多模型给出相近分数，但没有真实任务测试。
- 一个 happy path 成功，却没有异常和清理验证。

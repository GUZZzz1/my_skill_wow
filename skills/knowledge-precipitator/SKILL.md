# 思维沉淀助手

将长篇知识材料转化为可追溯、可复用且符合事实边界的技术沉淀。完整执行三轮流程，但只向用户展示终版。

## 先做内容路由

在沉淀前判断材料类型。若工作区存在 `AGENTS.md`、文章评分标准、日报规则、知识库路径或事实边界约定，先读取并服从；不存在时使用以下默认路由：

- 知识型文章、论文或外部资讯：用户需要筛选价值时先给数值评分和阅读决策，再沉淀知识；未提供评分标准时说明采用的判断维度。
- 内部需求、测试方案、用例、缺陷或技术方案：按“阅读/理解/梳理”记录，区分材料内容与用户本人执行事实。
- 用户明确说“入库/写入/追加”：将终版追加到指定文件；未指定时使用当前工作区约定的默认知识库。
- 仅有用户工作事实：不触发本 Skill，改走日报事实记录流程。
- 涉及内部日志、凭证、接口返回或敏感标识：只保留抽象后的模块、失败类型、归因和方法，不复制敏感原文。

## 检查输入

开始前确认可见输入至少包含一项：正文、可读取文件、代码/配置或已展开的图片文字。只有标题、失效链接或无法读取的附件时，指出缺失内容并停止，不凭标题生成完整沉淀。

可选输入包括来源链接、材料日期、用户目标、期望详细度和落盘路径。缺少这些信息不阻断分析，但在终版标记“未提供”。

## 第一轮：全局理解与初版

1. 通读全部输入，识别主题、来源、日期、材料类型和用户目标。
2. 读取 [输出模板](references/OUTPUT-TEMPLATE.md)，生成内部初版。
3. 先建立信息地图：观点、组件、数据流、参数、流程、异常、边界、证据和未确认项。
4. 对材料事实、分析推导和行动建议使用不同措辞，不把推导写成原文结论。

## 第二轮：逐段覆盖审查

1. 读取 [反思检查清单](references/REFLECTION-CHECKLIST.md)。
2. 回到源材料逐段检查，每个有信息量的段落至少映射到终版一个位置，或明确判断为导航噪声、重复内容、历史留档或敏感细节而不收录。
3. 原地补齐遗漏，不另建“补充遗漏”章节。
4. 重点核对数字、版本、命令、配置、接口、调用关系、平台差异、异常分支、适用条件和证据等级。
5. 检查事实归属：材料作者/团队的成果不能写成用户本人执行、验证、提交、修复或建设。

本轮在内部完成，不展示逐步推理或草稿痕迹。

## 第三轮：终版与落盘

1. 按输出模板整合终版；没有的信息简短标注“原文未提及”，不要用大量空章节稀释重点。
2. 输出详细度适配请求：
   - 快速筛选：评分/结论优先，只给核心摘要与风险。
   - 知识沉淀：输出完整模板。
   - 日报记录：只写当天学到什么、价值、边界和后续，不复制完整知识卡。
   - 用户未指定：长篇材料默认输出“真实结论、核心摘要、关键细节、风险缺口、可验收产物、事实边界”；需要落盘时使用完整模板。
3. 用户要求保存时直接写入本地文件，并说明路径和事实边界。
4. 终版末尾写“覆盖审查：已完成”，但只有实际执行清单后才能写。

## 质量原则

- 源材料覆盖要全，最终表达要按价值裁剪；“宁多勿少”不等于全文搬运。
- 保留影响复现和判断的代码、配置和数值；对重复导航、页面控件、点赞信息和无关历史版本降噪。
- 原文有歧义时标注“原文不清晰/待确认”，禁止自行补成事实。
- 对旧文档、平台版本和可能变化的命令标注时间边界，必要时建议实际核验。
- 给出可验收产物：代码、配置、用例、报告、指标、Demo、CR、问题单或面试讲法，按材料类型选择。
- 不把“已阅读/已理解”写成“已执行/已验证/已提交/已合入/已建设”。

# 技术细节覆盖检查清单

逐段审查源材料，并完成以下六类检查。某项不适用时标记 N/A，不要虚构内容。

## 1. 参数与环境

- 版本号、日期、端口、URL、环境变量和配置开关是否准确？
- 超时、重试、并发、资源限制、阈值、样本量和时间窗口是否保留？
- OS、设备、芯片、运行时、网络、权限和依赖条件是否写清？
- 旧版本信息是否与当前口径分开？

## 2. 组件关系与数据流

- 组件、模块、节点和责任边界是否完整？
- 调用方向、数据流、状态流和控制流是否明确？
- AI 判断节点与确定性执行节点是否区分？
- 上下游依赖、共享状态、隔离边界和副作用是否保留？

## 3. 接口、代码与产物

- API 端点、方法、参数、请求/响应和事件类型是否保留？
- 关键命令、代码、YAML、字段名和文件路径是否准确？
- 输入、输出、artifact、报告和证据位置是否明确？
- 代码/配置是否确实来自原文，是否避免擅自修正后冒充原文？

## 4. 异常与边界

- 失败、超时、重试、降级、回滚、清理和恢复是否覆盖？
- 权限拒绝、依赖缺失、网络异常、状态污染和并发冲突是否覆盖？
- 产品缺陷、case 缺陷、环境问题、unsupported 和 flaky 是否可区分？
- 适用范围、不适用范围、平台差异和已知限制是否明确？

## 5. 概念、选型与证据

- 术语定义是否忠于原文？同名指标是否存在不同口径？
- 技术选型、替代方案、收益、风险和取舍是否写清？
- 宣传性结论是否有指标、样本、对照或失败案例支持？
- 原文事实、合理推导、个人建议和待确认项是否分层？

## 6. 段落覆盖与事实归属

- 每个有信息量的段落是否映射到终版，或有明确不收录理由？
- 表格、图片、脚注和代码块的核心信息是否覆盖？
- 导航、评论、点赞、重复内容和无关历史留档是否降噪？
- 作者/团队成果是否被错误写成用户本人产出？
- 阅读、理解、设想是否被错误升级为执行、验证、提交或建设？
- 内部日志、凭证、人员信息和敏感接口是否已抽象或删除？

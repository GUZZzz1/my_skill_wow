# 仓库维护测试样例

## 应触发

1. “把这个新 Skill 加入 `GUZZzz1/my_skill_wow`，校验、更新 README 并发布。”
2. “全仓检查 Skill 的元数据、引用、资产、安装命令和远程状态。”
3. “修改这个 Skill 的触发说明和参考文件，复评后更新仓库。”

## 不应触发

1. “用已安装的周报 Skill 生成一份 HTML。”应由目标 Skill 执行，不是仓库维护。
2. “只评估这个 Skill 能不能准入，不修改和发布。”应使用 `evaluate-skill-quality`。
3. “帮我安装一个 GitHub 上的 Skill。”应使用 `skill-installer`。
4. “删除、弃用或重命名这个已发布 Skill。”超出本 Skill 的新增/修改范围，应停止并单独确认迁移方案。

## 正常 case

### 新增 Skill

输入：新 Skill 的目标、目标用户和仓库路径。

验收：

- Skill 由 `skill-creator` 初始化，目录和 `name` 一致。
- 直接引用文件完整，结构校验通过。
- `evaluate-skill-quality` 至少输出 `DESIGN_PASS`，没有 P0；只有满足相应工程门和证据等级时才能声称 `TEST_PASS` 或 `RELEASE_READY`。
- README 有目录、源码链接、Codex 提示词和终端命令。
- 临时安装目录中只出现目标 Skill。
- 推送后远程 commit ID 与本地一致。

### 兼容更新

输入：已发布 Skill 的局部修改。

验收：

- 确认输入、输出、`name` 和安装路径未破坏。
- 只重跑受影响和代表性 case，但不略过结构、隐私和引用检查。
- 已安装同名 Skill 不被静默覆盖。
- 提交中不包含无关更改。

## 失败与边界 case

1. **远程分支领先或分歧**：停止推送，报告差异，不强制覆盖。
2. **存在无关本地改动**：仅 stage 范围内文件；无法分离时向用户确认。
3. **发现凭据/隐私/私人原始材料**：停止发布，清理后重新扫描，报告时不回显完整敏感值。
4. **新 Skill 状态为 `NEEDS_REVISION` 或 `BLOCKED`**：不公开发布，将 P0/P1 转为可验收修改任务并复评。
5. **请求删除、弃用、重命名或回退**：明确说明超出范围，不在新增/修改流程中执行。

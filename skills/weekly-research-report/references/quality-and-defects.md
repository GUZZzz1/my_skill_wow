# 质量检查与缺陷处理

## D1-D9 检查

本地原有 `skill-evaluator` 是 8 大维度。本 Skill 采用扩展版 9 维，第 9 维是“评测闭环 / 可验证性”。不要把它说成官方原生九维。

| 维度 | 评估重点 |
|---|---|
| D1 元数据质量 | `name/description` 是否精准，能否正确触发，是否有反向排除 |
| D2 执行引导清晰度 | 触发后是否知道怎么做、何时追问、何时停止 |
| D3 领域知识密度 | 是否包含通用模型没有的专业知识、流程、判断标准 |
| D4 工作流完整性 | 是否覆盖端到端流程、分支、异常、失败兜底和迭代 |
| D5 输入输出清晰度 | 输入契约、输出格式、边界条件是否明确 |
| D6 资源利用 | 是否合理使用 `references/examples/scripts`，避免全塞进 `SKILL.md` |
| D7 写作质量 | 结构是否清晰、可扫读、无废话、无歧义 |
| D8 范围与聚焦 | 是否只解决一类稳定问题，避免过宽、过散、过窄 |
| D9 评测闭环 / 可验证性 | 是否有应触发/不应触发样本、失败 case、评分和迭代机制 |

## 用户反馈后的处理

用户说“内容混乱”：

- 先删图，不先加内容。
- 把节点进度和下周优先级放到工作概览。
- 每个章节只保留一个主要目的。

用户说“官话、看不懂”：

- 改成“事实 + 下一步 + 风险”。
- 删除抽象词。
- 保留必要术语，但第一次出现时解释它在本报告里的作用。

用户说“时间轴不对”：

- 重新问每项工作的起止日期。
- 区分审稿/返修、实验、写作三类工作。
- 日期不要都挤在同一行，必要时分上下两行或改成泳道图。

用户说“图不好看”：

- 先判断图是否必要。
- 对计划类内容优先用时间轴和节点图。
- 对工作量只做相对趋势，不用过硬百分比。

用户说“Skill 有缺陷”：

- 不直接重写整个 Skill。
- 按 D1-D9 定位问题。
- 小改一处，跑一组触发/不触发样例验证。

## 触发样例

应触发：

- “帮我生成这周研究生工作报告 HTML。”
- “按导师要求写一份论文周报，要有时间轴和图表。”
- “根据我的小论文投稿状态和大论文计划，生成周报。”

不应触发：

- “帮我润色论文摘要。”
- “检查 Word 参考文献格式。”
- “做一个普通前端 dashboard。”
